## 📌 TL;DR（执行摘要）

- **核心目标**：为不愿付费、追求音乐"拥有感"的技术型用户，打造一个集"跨平台搜索→下载→元数据补全→播放→管理"于一体的免费个人音乐库工具，定位为"个人音乐图书馆"而非流媒体App。
- **关键决策**：采用 Meting（5平台聚合搜索/下载/歌词/封面）+ MusicBrainz（深度元数据合规层）+ Last.fm（封面兜底/推荐）三层架构；技术栈推荐 Python + CustomTkinter（用户已有经验）；MVP 聚焦搜索→下载→播放→收藏→歌单核心闭环。
- **下一步**：暑假第1周（7/7–7/13）完成 M0 技术验证（Meting 可用性确认），这是整个项目的 Go/No-Go 决策点；8月中旬交付 MVP v1.0 作为暑假变现核心交付物。
- **最大风险**：免费音频获取是唯一无法合规解决的环节（依赖平台逆向接口），已明确标注合规边界为"个人自用可容，分发即越线"。

## 1. 产品目标（3个清晰、正交的目标）

MelodyHub 是一款面向个人用户的本地音乐库管理工具，定位为"个人音乐图书馆"而非流媒体播放器。以下三个目标分别覆盖**获取→管理→体验**三个维度：

| 编号 | 目标 | 衡量标准 |
|------|------|----------|
| **G1** | **跨平台聚合获取**：用户输入一个关键词即可搜索全网音乐资源并下载到本地，打破平台版权割裂 | 搜索结果覆盖≥3个音源平台；单一关键词的Top-5结果命中率≥80% |
| **G2** | **元数据完整性**：下载的每首歌曲自动附带封面、歌词、演唱者、作词/作曲等完整元数据，构建可检索的个人音乐资产库 | 下载歌曲的封面获取率≥90%、歌词获取率≥75%、作词/作曲填充率≥60% |
| **G3** | **纯净本地播放体验**：零广告、零推荐干扰、后台稳定播放，支持歌单管理与收藏，作为学习/编码场景的背景音乐 | 后台播放连续运行≥2小时无中断；应用启动到可播放≤3秒 |

**设计哲学**：用技术对抗付费墙与平台割裂——用户拥有自己下载的每一首歌的本地永久副本，元数据完整、无DRM、无广告。

## 2. 用户故事（3-5个场景）

> 来源：瑞思（用户研究员）

### US-1：搜索下载（主场景）
> 作为一个独立开发者，我在学习或编码时想听 Alan Walker 的《Faded》，我希望在 MelodyHub 中输入"Faded"或"Alan Walker"，看到来自多个平台的搜索结果，选择一首一键下载到本地，下载过程中自动获取封面、歌词和艺术家信息——整个过程不需要打开任何其他App，不需要看广告，不需要付费。

### US-2：元数据自动补全（体验分水岭）
> 作为一个对音乐有基本了解的听众，我下载了周杰伦的《晴天》，希望文件自带高清封面图、完整歌词、演唱者（周杰伦）、作词（周杰伦）、作曲（周杰伦）等信息——而不是一个只有文件名的裸文件，这样我在本地库浏览时能像用正规音乐App一样有视觉和信息的完整体验。

### US-3：歌单与背景播放（高频场景）
> 作为一个需要长时间专注学习的学生，我希望创建一个"学习BGM"歌单，把喜欢的欧美电音和华语流行都放进去，设置随机播放，然后切到后台让它一直放——不打断我的学习，不需要每隔几分钟去切歌。

### US-4：快速收藏
> 作为一个重度音乐消费者，我在浏览本地库时听到一首特别喜欢的歌，希望一键点击收藏按钮将它加入"我喜欢"列表，以后可以快速从收藏列表中找到它，而不需要记住歌名去搜索。

### US-5：本地库管理
> 作为一个音乐收藏者，随着下载的歌曲越来越多，我希望能按歌手、专辑浏览我的音乐库，能搜索某首歌，能对歌单里的歌曲重新排序——像一个真正的个人音乐图书馆，而不是一个混乱的文件夹。

## 3. 用户研究洞察（来自瑞思）

### 目标用户画像

| 维度 | 描述 |
|------|------|
| **身份** | 技术型个人用户——独立开发者 + 重度音乐消费者 |
| **画像** | 初中生，有 Godot + Python 开发经验，技术能力强 |
| **付费意愿** | 零——核心动机是"用技术对抗付费墙" |
| **音乐偏好** | 欧美流行（Alan Walker《Faded》）+ 华语流行（周杰伦《晴天》） |
| **核心场景** | 学习/编码时的后台背景播放（伴随型使用，非沉浸型） |
| **设备** | PC桌面端 |

### 核心痛点（按严重程度排序）

| 痛点 | 描述 | 严重度 |
|------|------|--------|
| **P1** | 付费墙越来越高（周杰伦全需付费） | 🔴 高 |
| **P2** | 广告干扰严重（免费听歌需忍受大量广告） | 🔴 高 |
| **P3** | 平台割裂需装多个App（没有单一平台能听全所有歌） | 🟡 中 |
| **P4** | 元数据缺失（下载的裸文件体验差，无封面无歌词） | 🟡 中 |
| **P5** | 数据不可控（平台随时下架，收藏的歌可能突然消失） | 🟡 中 |

### 6条关键洞察

1. **核心质量指标**：搜索下载成功率是用户留存的生命线——如果搜不到/下不下来，一切都无意义
2. **MVP范围应克制**：用户本身是开发者，理解分阶段交付，不宜一次性铺太大
3. **元数据是体验分水岭**：有封面有歌词 vs 裸文件是"正规App"和"文件夹"的区别，应紧随MVP实现
4. **免费方案可行性是最大风险**：PRD必须明确标注风险等级和合规边界
5. **音乐口味影响技术方案**：欧美靠MusicBrainz/Last.fm（合规），华语靠国内平台API（Meting），需多源整合
6. **后台稳定性 > 音质**：场景以后台背景播放为主，播放不中断比无损音质更重要

### 差异化定位

**"个人音乐图书馆"** 而非 "又一个流媒体App"。

五大差异化支柱：
1. **跨平台聚合搜索**——大厂受版权约束不可能做，这是结构性壁垒
2. **DRM-free本地永久拥有**——与所有竞品的订阅制商业模式根本对立
3. **元数据完整**——通过MusicBrainz补全竞品普遍缺失的作词/作曲信息
4. **零广告纯净体验**——没有商业变现压力
5. **个人化资产管理**——数据完全由用户掌控，平台下架与我无关

### 推荐技术架构：三层组合

| 层 | 方案 | 职责 |
|----|------|------|
| 数据获取层 | Meting（自建Node.js，v1.6.1活跃维护） | 搜索/播放链接/下载/歌词/封面，覆盖网易云+QQ+酷狗+酷我+百度五大平台 |
| 元数据增强层 | MusicBrainz API + Cover Art Archive | 补全作词/作曲/ISRC/发行年份/专辑信息（合规，ODbL协议，1 req/s限流需缓存） |
| 封面/推荐层 | Last.fm API | 封面兜底+相似艺术家推荐（免费API Key） |

**竞品不可模仿的壁垒**：跨平台聚合（大厂受版权约束不可能做）+ 本地永久库（与订阅制商业模式对立）。这个定位空间是竞品主动让出的。

### 市场数据

| 指标 | 数据 | 对MelodyHub的启示 |

| 中国数字音乐市场规模 | 2,113亿元 | 市场巨大，但MelodyHub不参与商业化 |

| 高频场景 | 通勤/运动/家务 | 离线播放是刚需（本地下载满足） |
| 汽水音乐增长 | +78% YoY | 短视频音乐平台在崛起，窗口期有限 |

### API可行性评估

| 能力 | 可行性 | 方案 | 风险等级 |
|------|--------|------|----------|
| **音乐搜索** | 🟢 中高 | Meting多平台 + MusicBrainz（欧美） | 低 |
| **音乐下载** | 🟡 中 | Meting（逆向接口，无合规源） | 🔴 高 |
| **封面获取** | 🟢 高 | Last.fm + Cover Art Archive（合规） | 低 |
| **歌词获取** | 🟡 中 | Meting/go-music-dl（逆向接口） | 🟡 中 |
| **艺术家/元数据** | 🟢 高 | MusicBrainz + Last.fm（合规） | 低 |

### 关键风险矩阵

| 风险 | 等级 | 描述 | 缓解措施 |
|------|------|------|----------|
| API封禁 | 🔴 高 | NeteaseCloudMusicApi（3万Star）2024年被清空，接口随时可能失效 | 多源fallback（5平台）；接口模块化设计，可快速替换 |
| 版权法律 | 🔴 高 | 个人自用有合理使用空间，分发即越线 | 明确定位为"个人本地工具"，不对外分发，不提供分享功能 |
| 接口变动维护 | 🟡 中 | 平台改版导致接口失效 | 抽象数据获取层，接口实现与业务逻辑解耦 |
| 欧美歌曲覆盖 | 🟡 中 | 小众歌曲可能"有元数据无音频" | MusicBrainz覆盖元数据 + 多平台fallback覆盖音频 |
| 免费方案可持续性 | 🟡 中高 | 音频获取环节可持续性最差 | 这是结构性风险，PRD中明确标注为已知风险 |

### 5条关键数据结论

1. 市场存在差异化切入空间，但非蓝海，窗口期有限——需尽快交付MVP
2. **免费音频获取是唯一无法合规解决的环节——核心风险**，需明确标注
3. 元数据层可构建"合规+风险"双层架构（MusicBrainz/Last.fm合规层 + Meting风险层）
4. 个人使用合规边界："自用可容，分发即越线"——产品定位为个人本地音乐管理工具
5. 欧美+华语双需求需API组合策略——元数据优先走合规源，音频获取设计多源fallback

## 6. 需求池（P0/P1/P2优先级）

### P0 — MVP（Must Have）

| 编号 | 需求 | 优先级 | 验收标准 | 估算(SP) |
|------|------|--------|----------|----------|
| REQ-001 | **跨平台音乐搜索** | P0 | Given 用户在搜索框输入关键词 When 点击搜索 Then 返回≥3个平台的搜索结果，每条结果显示[平台来源/歌名/艺术家/时长/封面缩略图]，结果按相关度排序，搜索响应≤5秒 | 5 |
| REQ-002 | **音乐下载到本地** | P0 | Given 用户在搜索结果中选择一首歌并点击下载 When 下载开始 Then 文件保存到本地指定目录（默认 ~/MelodyHub/Music/），下载进度可见，下载完成后文件可被本地播放器识别，支持MP3/FLAC格式 | 3 |
| REQ-003 | **本地音乐播放器** | P0 | Given 本地库中存在已下载的歌曲 When 用户双击或点击播放按钮 Then 音频开始播放，播放进度条可见且可拖拽，显示当前播放歌曲信息（封面/歌名/艺术家），支持后台播放≥2小时无中断 | 5 |
| REQ-004 | **播放控制** | P0 | Given 正在播放歌曲 When 用户点击上一首/下一首/暂停/播放 Then 播放器正确响应控制操作，切歌时无>1秒卡顿，播放队列保持连贯 | 2 |
| REQ-005 | **本地音乐库浏览** | P0 | Given 本地库中有≥1首歌曲 When 用户打开"音乐库"页面 Then 以列表/网格形式展示所有歌曲，每项显示[封面/歌名/艺术家/专辑/时长]，支持滚动浏览，点击可播放 | 3 |
| REQ-006 | **基础元数据自动获取** | P0 | Given 用户下载一首歌曲 When 下载完成 Then 自动获取并写入：封面图（获取率≥90%）、演唱者（≥95%）、歌词（≥75%）、歌名、专辑名。元数据写入文件ID3标签且同步存入SQLite库 | 4 |
| REQ-007 | **收藏歌曲** | P0 | Given 用户在播放或浏览时看到一首歌 When 点击收藏（♥）按钮 Then 该歌曲被标记为已收藏，"我喜欢"列表立即可见该歌曲，再次点击取消收藏 | 2 |
| REQ-008 | **创建与管理歌单** | P0 | Given 用户想创建歌单 When 点击"新建歌单"并输入名称 Then 歌单创建成功，可将歌曲添加到歌单，可从歌单移除歌曲，可在歌单中播放（自动加入播放队列） | 3 |

> **P0 合计：27 SP**

## 8. Non-goals（明确不做什么）

| 编号 | 不做的事项 | 理由 |
|------|-----------|------|
| NG-1 | **不做社交功能**（评论/分享/动态/关注） | 个人工具，无社交需求；社交会引入内容审核和合规风险 |
| NG-2 | **不做推荐算法引擎**（个性化推荐/每日推荐/算法歌单） | 与"纯净无干扰"定位冲突；推荐需要大量用户行为数据，个人工具不值得 |
| NG-3 | **不做音乐创作/编辑**（混音/剪辑/MIDI/录音） | 超出"音乐库管理"定位，技术栈不匹配 |
| NG-4 | **不对外分发/分享**（不提供P2P分享/上传/种子） | 合规红线——"自用可容，分发即越线" |
| NG-5 | **不做付费功能/商业化**（无内购/无广告/无会员） | 核心动机是"对抗付费墙"，商业化与定位根本对立 |
| NG-6 | **不做移动端App**（仅桌面端） | 用户场景是PC编码/学习，移动端开发成本对初中生过高 |
| NG-7 | **不做在线流媒体播放**（不做自己的流媒体服务） | 合规风险极高；定位是"下载到本地永久拥有" |
| NG-8 | **不做用户账号系统**（无登录/无注册/无云同步） | 个人本地工具，无需账号；云同步涉及服务器成本 |
| NG-9 | **不做音质增强/upscale**（不做AI升频/HIFI解码） | 后台播放场景音质非首要，技术复杂度过高 |
| NG-10 | **不做播客/有声书/视频** | 专注音乐库管理，不扩展内容类型 |

### 合规边界声明

> **"个人自用可容，分发即越线"**

MelodyHub 定位为**个人本地音乐管理工具**：
- ✅ 用户为自己使用而搜索、下载、管理音乐——属于个人合理使用范畴
- ✅ 元数据获取（MusicBrainz/Last.fm/Cover Art Archive）使用合规API（ODbL协议/免费API Key）
- ⚠️ 音频获取（Meting）依赖平台逆向接口，无合规免费源——这是已知结构性风险
- ❌ 不提供任何形式的对外分发、分享、传播功能
